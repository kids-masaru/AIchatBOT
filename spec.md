# Mora — システム仕様・改善方針書

最終更新: 2026-03-30

---

## 1. プロジェクト概要

**Mora（モラ）** は、個人事業主「モットラ」（masaru）が契約クライアント企業に提供する、LINE連携AIチャットボットサービスです。

- **提供形態**: SaaS型（1つのシステムで複数クライアントに提供）
- **エンドユーザー**: クライアント企業のお客様・従業員
- **AIエンジン**: Google Gemini（gemini-3-flash-preview）
- **チャネル**: LINE（Webhookによるリアルタイム応答）
- **デプロイ先**: Railway または Vercel

### サービスの価値

業務改善を専門とするモットラが、各クライアントの業務知識（マニュアル・FAQ・規約など）をGoogle Driveに登録するだけで、LINEでのお客様対応を自動化できる仕組みを提供する。

## 2. システムアーキテクチャ

```
[LINEユーザー]
     │ メッセージ送信
     ▼
[LINE Platform]
     │ Webhook (POST /webhook/{client_id})
     ▼
[Flask サーバー / app.py]
     │ クライアント認証・署名検証
     │ メッセージキューに積む（非同期）
     ▼
[Agent / core/agent.py]
     │ 会話履歴・ユーザープロフィール読み込み
     │ システムプロンプト構築
     ▼
[Google Gemini API]
     │ ツール呼び出し（最大10回ループ）
     ▼
[各種ツール]
     ├── Google Drive（ファイル検索・PDF読み取り）
     ├── Gmail（メール確認・下書き作成）
     ├── Google Calendar（予定確認・追加）
     ├── Web検索（DuckDuckGo）
     └── 計算・日付処理
     ▼
[LINE Push API]
     │ 回答をLINEで返信
     ▼
[LINEユーザー]
```

### 主要コンポーネント

| ファイル | 役割 |
|---|---|
| `app.py` | Flaskサーバー。LINE Webhookの受付、署名検証、クライアント振り分け、管理画面 |
| `core/agent.py` | AIの中核。Geminiへの問い合わせ、ツール実行ループ、会話履歴管理 |
| `core/prompts.py` | Moraの人格・行動指針・ツール定義 |
| `core/clients.py` | マルチテナント管理。クライアントごとのLINEトークン・設定を管理 |
| `core/communicator.py` | LINEへの返信送信 |
| `core/librarian.py` | Google Driveのファイル整理・検索担当（Akiという内部名） |
| `core/historian.py` | 会話履歴の管理 |
| `core/profiler.py` | ユーザープロフィールの生成・更新 |
| `core/scheduler.py` | 定期タスクのスケジューリング |
| `utils/vector_store.py` | Pineconeベクターデータベース（長期記憶） |
| `utils/sheets_config.py` | Google Sheetsからクライアント設定を読み込む |
| `dashboard/` | React製の管理画面（クライアントごとの設定UI） |

### 知識ソースの仕組み

```
[管理画面 Dashboard]
    │ Google DriveのフォルダIDを登録
    ▼
[Google Sheets（クライアント設定）]
    │ knowledge_sources として保存
    ▼
[Agent（Mora）]
    │ 質問に応じてフォルダ内を自動検索
    ▼
[Google Drive内のマニュアル・FAQ・規約PDFなど]
```

クライアントごとに「知識源フォルダ」を指定でき、そのフォルダ内のドキュメントをMoraが自動参照して回答する。

### マルチテナントの仕組み

```
LINE Webhook URL: /webhook/{client_id}

client_id = "client_A" → LINEトークンA + スプレッドシートA + 人格A
client_id = "client_B" → LINEトークンB + スプレッドシートB + 人格B
```

1つのサーバーで複数クライアントを管理。設定はGoogleスプレッドシートで一元管理。

## 3. 改善方針

### 背景

2026年3月のコードレビューにより、以下の問題が判明した。個人利用（Koto）から顧客提供サービス（Mora）への移行に際し、セキュリティ・品質面での整備が必要。

### 方針

- **Phase 1（必須）**: セキュリティと重大バグを修正してから顧客提供を開始する
- **Phase 2（推奨）**: 安定稼働のための品質改善
- **Phase 3（任意）**: 将来の拡張・運用効率化

---

## 4. セキュリティ要件

### 4.1 APIキー・トークン管理

- すべてのシークレット情報（LINEトークン、APIキー、パスワード）はサーバーの**環境変数**で管理する
- コードやJSONファイルにシークレットを平文で書いてはならない
- `master_config.json` はGitにコミットしない（.gitignoreに追加済みであることを確認）

### 4.2 認証

- LINE Webhookは必ず署名検証を行う（空の場合にスキップしない）
- 管理画面は強固なパスワードで保護する
- デバッグ用APIエンドポイントは本番環境では無効化する

### 4.3 マルチテナント分離

- クライアントAのデータがクライアントBに見えないよう、Pineconeのnamespaceをクライアントごとに分ける
- ユーザーIDの管理はリクエストスコープ内に閉じ、グローバル変数で共有しない

---

## 5. 品質要件

### 5.1 応答品質

- 知識源（Google Drive）に基づいた回答を優先する
- 不明な場合は「わかりません」と明確に答え、憶測で回答しない
- 応答時間：通常3〜10秒以内（ツール使用時は最大30秒）

### 5.2 信頼性

- LINEのWebhookは5秒以内に200を返す（非同期処理済み）
- エラー時はユーザーにわかりやすいメッセージを返す
- タイムアウトは全APIコールに設定する

### 5.3 クライアント設定の独立性

- 各クライアントの人格・知識・設定は完全に独立して動作する
- クライアントAの設定変更がクライアントBに影響しない

## 6. 将来の拡張方針

### 短期（Phase 1〜2完了後）

- クライアント数：5社程度まで安定稼働
- 知識ソース：Google Drive（マニュアル、FAQ、規約PDF）

### 中期

- 画像対応：LINEで送られた画像（チラシ、書類）を解析して回答
- 予約対応：Googleカレンダーと連携した予約受付・変更
- 定型返信：よくある質問への即時回答テンプレート

### 長期

- クライアント用セルフサービスダッシュボード（知識ソース登録をクライアント自身が管理）
- 月次レポート：問い合わせ傾向分析の自動レポート
